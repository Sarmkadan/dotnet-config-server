# ValidationRuleService

Service responsible for managing validation rules that can be applied to configuration values within the dotnet‑config‑server system. It provides CRUD operations for rules and a method to evaluate a configuration against the stored rules.

## API

### ValidationRuleService()
Initializes a new instance of the service. The constructor does not require any parameters and prepares the internal state for rule storage and retrieval.

### CreateRuleAsync
**Purpose:** Persists a new validation rule.  
**Parameters:** The data required to define the rule (rule definition).  
**Return Value:** A `Task<ValidationRule>` that completes with the created rule, including any server‑generated identifiers.  
**Throws:**  
- `ArgumentNullException` if the supplied rule definition is null.  
- `InvalidOperationException` if a rule with the same identifier already exists.  
- Any exception thrown by the underlying data store.

### GetRuleAsync
**Purpose:** Retrieves a single validation rule by its identifier.  
**Parameters:** The unique identifier of the rule to fetch.  
**Return Value:** A `Task<ValidationRule?>` that completes with the rule if found, or `null` if no rule matches the identifier.  
**Throws:**  
- `ArgumentException` if the identifier is empty or malformed.  
- Any exception thrown by the underlying data store.

### GetRulesAsync
**Purpose:** Retrieves all validation rules currently stored.  
**Parameters:** None.  
**Return Value:** A `Task<List<ValidationRule>>` that completes with a collection containing every rule. The list may be empty if no rules have been defined.  
**Throws:** Any exception thrown by the underlying data store.

### UpdateRuleAsync
**Purpose:** Modifies an existing validation rule.  
**Parameters:** The identifier of the rule to update and the updated rule definition.  
**Return Value:** A `Task<ValidationRule>` that completes with the rule after the update has been applied.  
**Throws:**  
- `ArgumentNullException` if the updated rule definition is null.  
- `KeyNotFoundException` if no rule exists with the supplied identifier.  
- `InvalidOperationException` if the update would cause a conflict (e.g., duplicate identifier).  
- Any exception thrown by the underlying data store.

### DeleteRuleAsync
**Purpose:** Removes a validation rule from storage.  
**Parameters:** The identifier of the rule to delete.  
**Return Value:** A `Task` that completes when the deletion operation finishes.  
**Throws:**  
- `ArgumentException` if the identifier is empty or malformed.  
- `KeyNotFoundException` if no rule exists with the supplied identifier.  
- Any exception thrown by the underlying data store.

### ValidateConfigurationAsync
**Purpose:** Evaluates a configuration value against all applicable validation rules and returns the result.  
**Parameters:** The configuration key and its associated value to validate.  
**Return Value:** A `Task<ValidationRuleResult>` that completes with the outcome of the validation, indicating success or any violated rules.  
**Throws:**  
- `ArgumentNullException` if the configuration key is null.  
- Any exception thrown during rule evaluation (e.g., formatting errors).

### Min
**Purpose:** Gets or sets the optional minimum value constraint for a rule.  
**Type:** `Nullable<decimal>`  
**Remarks:** When `null`, no minimum bound is enforced.

### Max
**Purpose:** Gets or sets the optional maximum value constraint for a rule.  
**Type:** `Nullable<decimal>`  
**Remarks:** When `null`, no maximum bound is enforced.

### OtherKey
**Purpose:** Gets or sets an optional secondary configuration key used in relational validation rules.  
**Type:** `string?`  
**Remarks:** When `null`, the rule does not reference another key.

### Operator
**Purpose:** Gets or sets the operator to apply when comparing values (e.g., `>`, `<=`, `==`).  
**Type:** `string?`  
**Remarks:** Expected to be a valid comparison operator supported by the validation engine.

### Message
**Purpose:** Gets or sets the custom message to return when the rule fails validation.  
**Type:** `string?`  
**Remarks:** If `null`, a default message is generated by the service.

## Usage

```csharp
// Example 1: Creating a rule and validating a configuration value
var service = new ValidationRuleService();

var rule = await service.CreateRuleAsync(new ValidationRule
{
    Key = "TimeoutSeconds",
    Operator = ">=",
    Min = 1,
    Message = "Timeout must be at least 1 second."
});

var result = await service.ValidateConfigurationAsync("TimeoutSeconds", 0);
if (!result.IsValid)
{
    Console.WriteLine(result.Message); // Outputs: Timeout must be at least 1 second.
}
```

```csharp
// Example 2: Updating a rule and retrieving all rules
var service = new ValidationRuleService();

// Assume a rule with id "maxConnections" exists
var updated = await service.UpdateRuleAsync("maxConnections", new ValidationRule
{
    Key = "MaxConnections",
    Operator = "<=",
    Max = 100,
    Message = "Connection count must not exceed 100."
});

var allRules = await service.GetRulesAsync();
foreach (var r in allRules)
{
    Console.WriteLine($"{r.Key}: {r.Operator} {r.Max ?? (object)r.Min}");
}
```

## Notes

- The service does not enforce any locking; concurrent calls to mutating methods (`CreateRuleAsync`, `UpdateRuleAsync`, `DeleteRuleAsync`) may result in race conditions. Callers should synchronize access if multiple threads may modify the rule set simultaneously.  
- Read‑only operations (`GetRuleAsync`, `GetRulesAsync`, `ValidateConfigurationAsync`) are safe to invoke concurrently with each other and with write operations, but the returned data reflects the state of the store at the moment the operation completes.  
- Nullable properties (`Min`, `Max`, `OtherKey`, `Operator`, `Message`) are intentionally nullable to allow flexible rule definitions; validation logic should treat a `null` value as “no constraint” for the corresponding field.  
- Passing an empty or whitespace‑only string for identifiers in `GetRuleAsync` or `DeleteRuleAsync` will cause an `ArgumentException`.  
- The service does not automatically persist changes; each method call interacts directly with the underlying storage, so a failure mid‑operation leaves the store in the state dictated by the storage implementation.  
- When using `ValidateConfigurationAsync`, if multiple rules apply to the same key, the result aggregates failures; the exact aggregation strategy is defined by the service’s internal validation pipeline.
